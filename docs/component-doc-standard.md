# FAUI 组件文档书写规范 (Component Documentation Standard)

本规范提炼自优秀的结构化文档（如 `box.md`），用于指导后续所有 FAUI 组件文档的编写与调整。请在新增或重构组件文档时，严格遵循本范式。

## 0. 编写前置要求与核心原则 (Prerequisites)

在开始编写任何组件文档之前，**严禁凭空臆想**组件的功能与属性。作者必须严格遵循以下原则：

- **阅读源码与类型**：必须深入阅读组件对应的源码实现（如 `src/components/xxx.tsx`）以及完整的 TypeScript 类型定义（如 `src/types/components/xxx.ts`）。
- **完整理解逻辑**：在完全理解组件的底层渲染逻辑、数据绑定机制和特有行为之后，再开始落笔编写文档。
- **明辨实现边界**：必须看清哪些 Ant Design 的原生属性已经被 FAUI 代理和实现，哪些暂未实现。文档中只能记录**已经明确实现并支持**的属性与功能，绝对不能直接照搬第三方官方文档或凭空臆测未实现的功能。
- **Action 配置真实性**：在所有 JSON 代码片段中，配置的动作（Action）必须是 FAUI 引擎真实存在的。严禁生搬硬套或凭空创造伪动作（如错误地将 `http_proxy` 臆想为 `http_request`，将 `message` 臆想为 `show_message`）。所有的 Action 配置必须遵循 `src/types/schema.ts` 中的 `ActionType` 联合类型和 `ActionPayload` 接口定义（例如 `message` 需要 `payload: { type, content }`，`http_proxy` 需要 `payload: { http_config }`）。

## 1. 整体结构骨架

一个标准的组件文档应该严格包含以下几个一级/二级标题（可根据组件特性增删部分区块）：

1. `# [组件名] 组件`（标题与简介）
2. `## 适用场景`
3. `## 核心属性`
4. `## [特有机制/高级用法]`（如：数据源绑定、子组件嵌套等，视组件而定）
5. `## 完整示例`
6. `## 新手常见问题` (FAQ)

## 2. 各模块内容书写细节

### ① 标题与简介 (`# [组件名] 组件`)
- **格式**：全小写组件名 + " 组件"，如 `# box 组件`。
- **内容**：用 1-2 句话概括组件的作用，可以类比 HTML 标签或 Ant Design 的原生组件，让开发者快速建立心智模型。

### ② 适用场景 (`## 适用场景`)
- **格式**：无序列表 (`-`)，每项加粗核心词。
- **内容**：列出 3-4 个该组件最典型的业务使用场景，帮助开发者判断“什么时候该用它”。
- **示例**：`- **页面根容器**：包裹整个页面的所有组件`

### ③ 核心属性 (`## 核心属性`)
- **属性总览**：首先，必须以表格或列表形式完整列举当前组件支持的所有属性（包含属性名、类型、默认值及简要说明）。
- **格式**：接着，按照属性的重要性进行降序排列，每个重要属性使用 `### [属性名]（[中文说明]）` 作为三级标题。
- **说明与表格**：对属性进行简要文字说明。如果属性是枚举值（如 `start`, `center`, `end`），**必须使用 Markdown 表格**列出：`| 值 | 效果 | 典型用途 |`。
- **代码片段**：每个核心属性下，**必须紧跟一个最小化、可运行的 JSON 代码片段**（使用 ` ```json ` 包裹），直观展示该属性如何配置。

### ④ 特有机制 / 高级用法 (`## [特有机制]`)
- **内容**：针对该组件独有的复杂逻辑进行展开说明。例如 `box` 的嵌套使用、`table` 的列定义 (`columns`) 和 `rowKey` 机制（例如：`table` 必须显式指定 `rowKey` 属性以避免 React 的 Key 警告并优化渲染性能，该值应对应数据源中每行的唯一标识字段，默认为 `'id'`）、表单组件的 `value.path` 双向绑定等。
- **图示辅助**：如果是复杂的嵌套关系，鼓励使用纯文本树形图辅助说明（如 `root ├── header └── content`）。

### ⑤ 通用能力整合（可选）
- **内容**：展示组件如何与 FAUI 的全局通用能力结合，例如 `## visible 条件显示`、`## 动态样式 (style)` 或表达式求值插值。

### ⑥ 完整示例 (`## 完整示例`)
- **内容**：提供一个贴近真实业务场景的、组合了多种属性的**完整 JSON 配置片段**。最好是一个完整的数组 `[...]` 或包含上下文的结构，方便开发者直接 Copy & Paste 到本地预览。

### ⑦ 新手常见问题 (`## 新手常见问题`)
- **格式**：采用 Q&A 问答形式。
  - 问题加粗：`**Q: [常见症状描述]？**`
  - 答案列表：`- [排查步骤或解决方案]`
- **内容**：主要收录容易配错的属性、不符合预期的渲染表现（如高度不撑开、数据没回写、组件未渲染等防呆指南）。
  - 例如：`button` 没有文字（因为使用了 `content` 而不是 `label`）。
  - 例如：`checkbox` 数据绑定未生效（因为使用了 `value` 而不是推荐的 `checked`）。
  - 例如：`table` 报 React 的 Key 警告（因为没有显式配置 `rowKey` 属性或数据源缺少 `id` 字段）。
- **要求**：至少提供 2-3 个真实痛点。**必须根据源码推导**，严禁生搬硬套。

## 3. 排版与排版约定

- **代码高亮**：所有的组件名、属性名、变量名、特定值，必须使用反引号 \` 包裹（例如 `layout`、`vertical`）。
- **JSON 规范**：文档中的 JSON 必须是合法的 JSON 格式（双引号包裹键名），可以带有 `//` 注释（为了阅读方便，假定阅读器支持 JSON5 或允许注释）。
- **重点强调**：对于容易踩坑的地方，使用 `**注意**：` 或 `> 💡 提示：` 进行高亮拦截。

## 4. 最终审查与验收标准 (Final Review)

- **合理性自查**：在完成文档初稿后，作者必须通读全文，重点检查文档书写的**合理性**。特别是“新手常见问题（FAQ）”和示例代码部分，要避免生搬硬套、前后矛盾或脱离实际业务场景。
- **验收确认**：绝不能写完即抛。只有在仔细排查并确认全文内容准确、逻辑自洽且无误导性信息之后，才能算作该组件的文档正式编写/重构完成。
